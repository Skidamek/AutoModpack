### Encryption

AutoModpack now prefers an iroh-based QUIC transport for modpack downloads.

The QUIC session handles encryption and peer authentication directly, while AutoModpack uses the server's endpoint ID as the primary user-visible identity.

When the server enables `shareMinecraftConnection`, AutoModpack can carry an Iroh custom transport over the existing Minecraft connection during the in-game update flow. This avoids opening a second TCP bootstrap socket in restrictive proxy, firewall, or NAT environments. Preload and offline update checks use the same Iroh download protocol, but only with the saved direct-IP and raw-TCP routes from a previous successful session.

On first connect, the client asks you to verify that endpoint ID unless the server published a matching DNSSEC TXT proof. Once trusted, the same server can be recognized on future connections. Read [how identity verification works](certificate) for the exact DNS record format and rules.

### Direct Downloads

AutoModpack uses direct download links from Modrinth and CurseForge APIs to download most of your modpack files.
This means that the mod authors get credit for every download.

However, if file you are providing in the modpack is not available on Modrinth or CurseForge, AutoModpack will download it (also directly) but from your server.

### Compression

To speed up the download, AutoModpack compresses the modpack files using [Zstandard](https://facebook.github.io/zstd/) compression algorithm.
This algorithm is very fast and provides great compression ratio, which means that the modpack files are smaller and faster to download.
This is especially useful for larger modpacks or slower internet connections.
However, this only applies to files downloaded from your server.

### Authorization

<Callout>
Do not confuse this with [server identity verification](certificate), which is used to verify the server's public key.
</Callout>

AutoModpack authorizes modpack downloads through the player's authenticated Minecraft identity and the player's bound Iroh endpoint/public key.

During Minecraft login, the client announces its Iroh public key over the encrypted Minecraft connection. The server binds that key to the authenticated player UUID. Later Iroh connections are accepted only when:

- the remote Iroh TLS handshake proves possession of the matching private key
- that endpoint ID is bound to a known player UUID
- that UUID is currently allowed by the server's whitelist / ban rules

In online mode this gives strong player-bound authorization. In offline mode the same logic still applies, but it inherits Minecraft's weaker identity model.
