### Choose a certificate verification method

AutoModpack always authenticates the modpack host as the **Minecraft server address selected by the player**. `addressToSend` and `portToSend` may route the TCP connection elsewhere, but that advertised route never chooses the trusted identity.

Use the first method that fits your server:

| Method | Best for | Saved by AutoModpack | Certificate rotation | Verification |
| --- | --- | --- | --- | --- |
| **CA-signed certificate** | Public servers with a domain; recommended default | No | Renew normally through the CA | Standard PKIX validity and Minecraft-hostname verification |
| **DNSSEC AMP1** | Self-signed certificates on a DNSSEC-enabled domain | No | Update DNS and allow for TTL propagation | DNSSEC fingerprint match plus certificate validity |
| **Pinned server address** | Private communities distributing an authenticated join address | Yes, per Minecraft `host:port` | Redistribute the new pinned address | Exact leaf-certificate fingerprint |
| **Manual verification / TOFU** | Last-resort setup without a CA, DNSSEC, or pinned address | Yes, per Minecraft `host:port` | Remove the old pin and verify the replacement | Exact leaf-certificate fingerprint after acceptance |

The trust order is deliberate:

1. Any pending or saved certificate pin is mandatory, whether it came from an address or a player decision. A mismatch always fails closed.
2. With no saved pin, AutoModpack tries normal CA/WebPKI verification.
3. A valid self-signed certificate may be authenticated through DNSSEC AMP1.
4. If no automatic method applies, AutoModpack asks the player and saves the accepted certificate as a pin.

### CA-signed certificate

A publicly trusted certificate provides the simplest player experience and is the recommended default for public servers. AutoModpack uses the standard Java trust store, checks the certificate validity period, and verifies it against the Minecraft hostname players entered.

You can obtain a certificate with tools such as [Certbot](https://eff-certbot.readthedocs.io/en/stable/install.html). Replace `cert.crt` with the full certificate chain and `key.pem` with the private key in `~/automodpack/.private/`. The private key must use [PKCS#8 PEM format](https://netty.io/wiki/sslcontextbuilder-and-private-key.html).

The certificate must be valid for the original Minecraft hostname. A different modpack endpoint hostname is only a TCP route and does not need to appear in the certificate.

#### Certbot example

```bash
# certbot certonly --manual --preferred-challenges dns -d <minecraft-server-domain>
```

<Callout variant="warning">
Never copy-paste random commands from the internet. Always read the documentation of the tool you are using.
</Callout>

Use `fullchain.pem` as `cert.crt` and `privkey.pem` as `key.pem`. Protect the private key carefully: access to it permits server impersonation.

### DNSSEC AMP1 for self-signed certificates

If you want to keep AutoModpack's self-signed certificate and control a DNSSEC-enabled domain, publish its fingerprint in DNS. Clients can then authenticate it automatically without storing a local pin or showing the first-connection prompt.

Run:

```text
/automodpack host fingerprint dns <minecraft-hostname>
```

Use the hostname players enter in Minecraft, including the public name that owns any Minecraft SRV record. The command prints a record like:

```dns
_automodpack.play.example.com. IN TXT "v=amp1;fp=<certificate-sha256>"
```

Publish exactly one AMP1 record in the DNSSEC-signed zone. [deSEC](https://desec.io/) is a strong free managed DNS option when your current DNS provider does not offer DNSSEC; you can host a domain or delegate a subdomain to it. AutoModpack does not depend on deSEC specifically—any correctly configured DNSSEC provider works.

When normal CA verification fails for a genuinely self-signed leaf, clients query both Cloudflare and Mullvad over DNS-over-HTTPS. The certificate is accepted only when both resolvers report DNSSEC validation, agree on the AMP1 fingerprint, the fingerprint matches, and the certificate is currently valid.

- A match authorizes that connection but is not saved locally.
- An authoritative mismatch, malformed record, or conflicting AMP1 records fail closed without a manual override.
- An absent or unavailable record falls back to manual verification.

When the certificate changes, update the single `fp` value and allow for DNS TTL and resolver-cache propagation before deploying the new certificate.

### Shareable pinned server addresses

For a simple authenticated copy/paste flow, generate both the normal address and a pinned AutoModpack address:

```text
/automodpack host fingerprint share <minecraft-address>
```

The pinned form looks like:

```text
play.example.com:25565#amp1=<certificate-sha256>
```

Supporting clients import the public fingerprint into AutoModpack's certificate trust store and save only the normal Minecraft address. Removing AutoModpack therefore leaves a usable vanilla server entry.

The imported fingerprint becomes the origin's certificate pin. The exact presented leaf certificate must match even if another certificate would otherwise be accepted by a CA or DNSSEC. A mismatch has no Verify or Skip bypass. To rotate the certificate, distribute a newly generated pinned address.

Metadata owned by another mod can coexist as another `#key=value` segment in either order. AutoModpack removes only its `amp1` segment and preserves the rest:

```text
play.example.com:25565#amp1=<fingerprint>#othermod1=<value>
play.example.com:25565#othermod1=<value>#amp1=<fingerprint>
```

The pinned address is only as trustworthy as the channel used to distribute it. Provide the plain address for vanilla and older clients.

### Manual verification and TOFU

When no automatic method is available, AutoModpack shows the presented fingerprint. Players can paste a fingerprint obtained from the server operator, or explicitly choose the higher-risk Skip flow. A successful decision becomes an exact certificate pin for that Minecraft `host:port`, using the same enforcement and storage as an imported address pin.

The acceptance reason is retained as `TOFU` or `ADDRESS_PIN` for UI and future migration purposes, but both behave identically after they are saved: the certificate must match before CA or DNSSEC is considered. A changed certificate requires removing/replacing the old pin or importing a newly pinned address.

Retrieve the current public fingerprint with:

```text
/automodpack host fingerprint
```

A fingerprint is public verification data, not a password. It protects clients from accepting an impersonating host; it does not grant access to the private key.

### Why certificate verification matters

If an attacker controls the AutoModpack connection, they can provide arbitrary mod files. Minecraft mods can access local files and execute arbitrary code, so authenticating the modpack host protects every connecting player.

Certificate verification does not replace secure server administration. Continue using strong passwords, two-factor authentication, restricted file permissions, current software, and careful private-key handling.
