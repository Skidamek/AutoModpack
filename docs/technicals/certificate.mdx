### Automatic verification for self-signed certificates with DNSSEC

AutoModpack normally verifies CA-signed certificates through the standard Java trust store, just like HTTPS. No DNSSEC record is needed for a valid CA-signed certificate.

When the modpack host uses AutoModpack's self-signed certificate, a server owner with a DNSSEC-enabled domain can publish its fingerprint in DNS. Clients then verify it automatically instead of showing the first-connection fingerprint prompt.

Run this command on the server:

```text
/automodpack host fingerprint dns <minecraft-hostname>
```

Use the hostname players enter in Minecraft, including the public name that owns any Minecraft SRV record. The command prints a record like:

```dns
_automodpack.play.example.com. IN TXT "v=amp1;fp=<certificate-sha256>"
```

Publish exactly that TXT record in the DNSSEC-signed zone. The record contains one SHA-256 fingerprint and does not contain a download address or port. `addressToSend` and `portToSend` still select the modpack endpoint; DNS only authenticates the self-signed certificate presented by that endpoint.

On each new AutoModpack connection, the client first attempts normal CA verification. DNS is queried only when that verification fails and the presented leaf certificate is genuinely self-signed. Its lifetime fields are not used for exact-fingerprint trust. Clients ask both Cloudflare and Mullvad over DNS-over-HTTPS and use the fingerprint only when both resolvers report DNSSEC validation and agree.

The resulting behavior is:

- a valid CA-signed certificate is accepted normally without using the DNS record;
- a self-signed certificate matching the DNSSEC fingerprint is accepted automatically for that connection and is not saved in the known-hosts file;
- an authoritative DNSSEC fingerprint mismatch or malformed AutoModpack record fails closed without a manual override prompt;
- when no usable record is available, AutoModpack falls back to the existing saved/manual fingerprint flow.

If the self-signed certificate changes, replace the single `fp` value with the new fingerprint. Allow for DNS TTL and resolver cache propagation when planning the change; clients seeing an old fingerprint will safely refuse the new certificate until the record has propagated.

### Manual fingerprint verification

When automatic CA or DNSSEC verification is unavailable, copy the fingerprint from the server console into the game. Manual verification and the explicit Skip action both save the presented fingerprint permanently for the original Minecraft hostname. A route or port change keeps working while the exact certificate remains unchanged; a certificate change requires another trust decision.

You can retrieve the current fingerprint at any time with:

```text
/automodpack host fingerprint
```

A fingerprint is public verification data, not a password. Anyone who can reach the modpack host may still be able to download public modpack data; the fingerprint protects clients from accepting an impersonating host. The fingerprint changes whenever the leaf certificate changes.

### Providing a CA-signed certificate

A publicly trusted certificate allows standard WebPKI verification without a DNSSEC AutoModpack record. You can obtain one with tools such as [Certbot](https://eff-certbot.readthedocs.io/en/stable/install.html).

After obtaining the certificate, replace `cert.crt` (the full certificate chain) and `key.pem` in `~/automodpack/.private/`. The private key must use [PKCS#8 PEM format](https://netty.io/wiki/sslcontextbuilder-and-private-key.html). Protect these files carefully: access to the private key permits server impersonation.

The certificate must be valid for the Minecraft hostname players enter. A different modpack endpoint hostname is only a TCP route and does not need to appear in the certificate.

### Certbot example

Install Certbot, then request a certificate for the Minecraft hostname players use:

```bash
# certbot certonly --manual --preferred-challenges dns -d <minecraft-server-domain>
```

<Callout variant="warning">
Never copy-paste random commands from the internet. Always read the documentation of the tool you are using.
</Callout>

The files needed by AutoModpack are `fullchain.pem` (copy and rename it to `cert.crt`) and `privkey.pem` (copy and rename it to `key.pem`).

### Why is this necessary?

If an attacker could take control of the AutoModpack connection, they could deliver arbitrary files to players. Minecraft mods can execute arbitrary code, so authenticating the modpack host is an important part of protecting every client that joins the server.

A private or little-known server is not automatically safe. Internet scanners can discover exposed Minecraft servers, and local DNS or network attacks can target players before they reach the intended server. Certificate verification does not replace secure server administration: continue using strong passwords, two-factor authentication, restricted file permissions, and up-to-date software.
